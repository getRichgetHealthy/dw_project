# 监管数据集市 - 数据模型设计

## 一、整体数据架构

```
数据源层(SRC) -> ODS层 -> STG层 -> DW层 -> DM层 -> APP层
```

## 二、ODS层（操作数据层）

### 2.1 设计原则
- 与源系统保持一致的结构
- 保留原始数据不做清洗
- 增量同步与全量同步相结合

### 2.2 表清单

| 表名 | 中文名 | 数据来源 | 更新频率 |
|------|--------|----------|----------|
| ods_customer | 客户信息原始表 | 核心系统 | 日 |
| ods_account | 账户信息原始表 | 核心系统 | 日 |
| ods_transaction | 交易流水原始表 | 核心系统 | 日 |
| ods_contract | 合同信息原始表 | 信贷系统 | 日 |
| ods_organization | 机构信息原始表 | 人力系统 | 月 |

## 三、STG层（临时层）

### 3.1 设计原则
- 数据清洗与标准化
- 统一数据编码
- 数据脱敏处理

### 3.2 表清单

| 表名 | 中文名 | 说明 |
|------|--------|------|
| stg_customer | 客户信息清洗表 | 标准化客户数据 |
| stg_account | 账户信息清洗表 | 标准化账户数据 |
| stg_transaction | 交易流水清洗表 | 标准化交易数据 |
| stg_contract | 合同信息清洗表 | 标准化合同数据 |

### 6.2 详细ETL流程

#### 6.2.1 ODS层数据加载
| 任务名称 | 数据源 | 目标表 | 加载方式 | 执行频率 |
|----------|--------|--------|----------|----------|
| ods_customer_full | 核心系统 | ods_customer | 全量 | 日 |
| ods_account_inc | 核心系统 | ods_account | 增量 | 日 |
| ods_transaction_inc | 核心系统 | ods_transaction | 增量 | 日 |
| ods_contract_inc | 信贷系统 | ods_contract | 增量 | 日 |

#### 6.2.2 STG层数据清洗
```sql
-- 客户信息清洗 (stg_customer)
INSERT INTO stg_customer
SELECT 
    customer_id,
    TRIM(customer_name) AS customer_name,
    id_type,
    id_number,
    CASE WHEN gender IN ('1','男') THEN 'M' 
         WHEN gender IN ('2','女') THEN 'F' 
         ELSE 'U' END AS gender,
    TO_DATE(birthday) AS birthday,
    nationality,
    occupation,
    address,
    phone,
    customer_type,
    customer_level,
    risk_level,
    TO_DATE(create_date) AS create_date,
    CURRENT_TIMESTAMP() AS etl_time
FROM ods_customer
WHERE is_valid = '1';
```

#### 6.2.3 DW层数据整合

**客户主题ETL流程：**
1. 从STG层抽取客户数据
2. 执行SCD(缓慢变化维度)处理
3. 关联客户关系信息
4. 生成客户风险评分
5. 写入DW层客户表

```sql
-- 客户维度全量同步
INSERT OVERWRITE TABLE dw_customer_dim PARTITION(dt='${bizdate}')
SELECT 
    c.customer_id,
    c.customer_name,
    c.id_type,
    c.id_number,
    c.gender,
    c.birthday,
    c.nationality,
    c.occupation,
    c.address,
    c.phone,
    c.customer_type,
    c.customer_level,
    COALESCE(r.risk_level, 'NORMAL') AS risk_level,
    c.create_date,
    CURRENT_TIMESTAMP() AS update_date,
    'Y' AS is_active,
    CURRENT_TIMESTAMP() AS etl_time
FROM stg_customer c
LEFT JOIN (
    SELECT customer_id, MAX(risk_level) AS risk_level 
    FROM stg_customer_risk 
    GROUP BY customer_id
) r ON c.customer_id = r.customer_id;
```

**交易主题ETL流程：**
1. 从ODS层抽取交易数据
2. 数据清洗与标准化
3. 识别大额交易(单笔≥50万)
4. 识别可疑交易(规则引擎)
5. 写入DW层交易表

```sql
-- 大额交易识别
INSERT INTO dw_large_trans_report
SELECT 
    CONCAT('LARGE_', trans_id) AS report_id,
    customer_id,
    customer_name,
    id_number,
    trans_amount,
    trans_date,
    trans_type,
    account_id,
    'LARGE' AS report_type,
    'PENDING' AS report_status,
    CURRENT_DATE() AS report_date,
    CURRENT_TIMESTAMP() AS etl_time
FROM stg_transaction
WHERE trans_amount >= 500000;
```

**风险主题ETL流程：**
1. 计算各类风险指标
2. 与阈值对比判定风险等级
3. 生成风险预警
4. 写入风险表

### 6.3 数据质量检查

| 检查类型 | 检查规则 | 处理方式 |
|----------|----------|----------|
| 完整性 | 主键唯一性检查 | 标记异常数据 |
| 完整性 | 空值检查 | 记录并告警 |
| 准确性 | 数据范围检查 | 标记异常 |
| 一致性 | 跨表一致性 | 记录差异 |
| 及时性 | 数据时效检查 | 告警 |

